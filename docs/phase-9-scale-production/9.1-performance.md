# 9.1 — Performance Optimization

## What We're Building

Making the app fast. Not "developer laptop fast" but "₹8,000 Android phone on slow 3G fast". The target users use budget smartphones with limited RAM and inconsistent network speeds.

## Performance Targets

| Metric | Target | How to Measure |
|--------|--------|---------------|
| First Contentful Paint (FCP) | < 1.5s | Lighthouse |
| Largest Contentful Paint (LCP) | < 2.5s | Lighthouse |
| Time to Interactive (TTI) | < 3.5s | Lighthouse |
| Total bundle size (initial) | < 200KB gzipped | Build output |
| Firestore reads per page load | < 5 | Firebase console |

## Optimization Areas

### 1. Bundle Size

**Problem**: Nuxt + Firebase + Tailwind can produce large bundles.

**Solutions**:

**Tree-shake Firebase**: Only import what you use.
```typescript
// GOOD: modular imports
import { getAuth } from 'firebase/auth'
import { getFirestore, collection, getDocs } from 'firebase/firestore'

// BAD: full imports
import firebase from 'firebase/app'
```

**Analyze bundle**: Use `nuxt analyze` to see what's large.

```bash
npx nuxi analyze
```

**Lazy load routes**: Nuxt does this by default with file-based routing. But verify heavy pages aren't in the initial bundle.

**Defer non-critical scripts**: Load Firebase Messaging and analytics only after the initial render.

### 2. Firestore Query Optimization

**Problem**: Too many reads = slow + expensive.

**Solutions**:

**Use composite indexes**: For common queries (orders by retailerId + status, products by category + active).

**Limit query results**: Don't fetch ALL orders for the dashboard. Fetch the most recent 50, paginate for more.

**Cache aggressively**: Firestore offline persistence (enabled in Phase 7) caches reads. Second page load is instant.

**Denormalize wisely**: Product names in orders, retailer names in orders — avoid extra reads just to display a name.

**Listen sparingly**: Real-time listeners on large collections are expensive. Use them only for:
- Wholesaler's order dashboard (live updates)
- Delivery boy's active orders
- Individual order detail pages

Don't listen to the entire product catalog — fetch once and cache.

### 3. Image Optimization (Already from Phase 8)

- Serve thumbnails for list views (200px)
- Lazy load all images
- Use WebP where supported
- Set long cache headers

### 4. Component-Level Optimization

**Use `v-once` for static content**: Parts of the layout that don't change.

**Use `shallowRef` for large lists**: Product catalog and order lists don't need deep reactivity.

```typescript
const products = shallowRef<Product[]>([])
```

**Virtualize long lists**: If the product list exceeds 100 items, use a virtual scroller so Vue only renders visible items. Library: `vue-virtual-scroller`.

**Debounce expensive computations**: Search, filter, and sort operations should be debounced.

### 5. SSR vs CSR Decision

Current approach: Client-side only (Firebase is client-side).

**Consider SSR for**:
- Landing page (if you build a public-facing marketing page)
- SEO (not needed for the app — it's behind auth)

**Keep CSR for**:
- All authenticated app pages (Firebase Auth doesn't work server-side without Firebase Admin)

For this app, CSR is correct. Don't add SSR complexity.

### 6. Prefetching

**Prefetch product data**: On login redirect, start fetching products before the page renders.

**Prefetch linked pages**: When hovering over a navigation link, prefetch that page's data. Nuxt handles route prefetching automatically.

## Measurement & Monitoring

### Lighthouse Audit

Run regularly:
```bash
npx lighthouse http://localhost:3000 --view
```

Or use Chrome DevTools → Lighthouse tab.

### Firebase Performance Monitoring (Optional)

```bash
npm install firebase/performance
```

Tracks:
- Page load times
- Network request durations
- Custom traces for specific operations

### Web Vitals

Monitor Core Web Vitals in production:

```typescript
import { onCLS, onFID, onLCP } from 'web-vitals'

onCLS(console.log)
onFID(console.log)
onLCP(console.log)
```

Send to analytics or log to Firestore for tracking.
