# 9.2 — Security Hardening

## What We're Building

A thorough security review and hardening of the entire system. Phase 0 set up basic security rules. This phase stress-tests them and closes gaps.

## Security Audit Checklist

### 1. Firestore Security Rules Audit

Review every rule written in Phase 0. Test with the Firebase Emulator Security Rules testing:

**Test cases to write**:
- Retailer CANNOT read another retailer's orders
- Retailer CANNOT modify products
- Retailer CANNOT change order status
- Delivery CANNOT read unassigned orders
- Delivery CANNOT modify order items
- Unauthenticated users CANNOT read anything
- Wholesaler CAN do everything

**Tool**: Firebase Emulator Suite includes a rules test runner:

```typescript
import { assertFails, assertSucceeds } from '@firebase/rules-unit-testing'

// Example test
test('retailer cannot read other retailer orders', async () => {
  const db = getFirestoreForUser('retailer1')
  await assertFails(db.collection('orders').doc('order-of-retailer2').get())
})
```

### 2. Input Validation

Every user input must be validated both client-side AND in Firestore rules:

| Input | Validation |
|-------|-----------|
| Phone number | Matches +91 followed by 10 digits |
| Quantity | Integer > 0, < 10000 |
| Price | Number >= 0, < 1000000 |
| Product name | String, 2-200 characters, no script tags |
| Order status | Must be one of the allowed values |
| Status transition | Must follow allowed transition map |
| Image upload | File type must be image/*, size < 5MB |

**Firestore rules for validation**:
```
allow create: if
  request.resource.data.items.size() > 0 &&
  request.resource.data.status == 'pending' &&
  request.resource.data.totalAmount >= 0;
```

### 3. XSS Prevention

**Risk**: User-generated content (product names, notes) could contain malicious scripts.

**Mitigation**:
- Vue's template system auto-escapes HTML by default (safe)
- Only use `v-html` with sanitized content
- The search highlight feature (Phase 4) uses `v-html` — sanitize the query first:
  ```typescript
  function escapeHtml(text: string): string {
    return text.replace(/[&<>"']/g, match => ({
      '&': '&amp;', '<': '&lt;', '>': '&gt;',
      '"': '&quot;', "'": '&#39;'
    }[match]!))
  }
  ```

### 4. Authentication Security

- **Session management**: Firebase Auth handles token refresh automatically
- **Force logout on block**: When a user is blocked, force-invalidate their session via Cloud Function (revoke refresh tokens)
- **Rate limiting**: Firebase Auth has built-in rate limiting for OTP requests
- **No sensitive data in localStorage**: Only Firebase Auth uses localStorage for session tokens (this is expected and managed by Firebase)

### 5. API Security (Cloud Functions)

If using Cloud Functions (Phase 7 notifications, Phase 8 AI):
- Verify caller is authenticated: `context.auth` check
- Validate all input parameters
- Set appropriate CORS headers
- Rate limit expensive operations (Vision API)

```typescript
export const someFunction = functions.https.onCall(async (data, context) => {
  if (!context.auth) {
    throw new functions.https.HttpsError('unauthenticated', 'Must be logged in')
  }
  // ... validate data ...
})
```

### 6. Environment Variable Security

- `.env` is in `.gitignore` (verified)
- Firebase config is public (API key is safe to expose — security comes from rules, not key secrecy)
- Cloud Function environment variables are set via `firebase functions:config:set`
- No secrets are hardcoded in source code

### 7. Dependency Audit

Run regularly:
```bash
npm audit
npm audit fix
```

Keep dependencies updated. Set up Dependabot or Renovate bot on GitHub for automatic PRs.

## Security Testing

### Manual Tests

1. Log in as retailer → try accessing `/wholesaler/dashboard` → should redirect
2. Open browser DevTools → try calling Firestore directly → should fail for unauthorized operations
3. Modify a request payload → Firestore rules should reject invalid data
4. Try SQL injection in search fields → should be harmless (Firestore isn't SQL)
5. Try XSS in product names → should be escaped in UI

### Automated Tests

Write Firestore security rules tests (using `@firebase/rules-unit-testing`) for all critical paths.
