# 1.2 — Authentication Flow

## What We're Building

Complete login system using Firebase Phone Auth with OTP. After login, the app determines the user's role and routes them to the correct dashboard.

## The Login Flow (Step by Step)

```
1. User opens app → redirected to /login
2. User enters phone number (Indian format)
3. App calls Firebase signInWithPhoneNumber()
4. Firebase sends OTP via SMS (or uses test number in dev)
5. User enters 6-digit OTP
6. App calls confirmationResult.confirm(otp)
7. Firebase Auth creates/verifies the user
8. App reads user document from Firestore (users/{uid})
9. IF document exists → read role → redirect to role dashboard
10. IF document DOESN'T exist → show "Contact your wholesaler" screen
```

## Work Items

### 1. Build Login Page

**File**: `pages/login.vue`

Two-step form:
- **Step 1**: Phone number input + "Send OTP" button
- **Step 2**: OTP input (6 digits) + "Verify" button

UI requirements:
- Indian phone format validation (+91 followed by 10 digits)
- Loading spinner on OTP send
- "Resend OTP" after 30 second countdown
- Error messages for invalid number, wrong OTP, expired OTP
- Disable button during network request (prevent double-tap)

### 2. Build Auth Composable

**File**: `composables/useAuth.ts`

Exposes:
- `sendOtp(phoneNumber)` — triggers Firebase phone auth
- `verifyOtp(code)` — confirms the OTP
- `logout()` — signs out
- `user` — reactive ref of current Firebase user
- `userProfile` — reactive ref of Firestore user document
- `role` — computed from userProfile
- `isAuthenticated` — boolean
- `isRetailer` / `isWholesaler` / `isDelivery` — role checks
- `loading` — auth state is being determined
- `error` — last auth error message

### 3. Build Auth Store

**File**: `stores/auth.ts`

Persists auth state across the app:
- Listens to Firebase `onAuthStateChanged`
- When user signs in, fetches their Firestore profile
- When user signs out, clears all state
- On app load, checks if there's an existing Firebase session

### 4. Build Auth Middleware

**File**: `middleware/auth.global.ts`

Runs on every route:
- If not authenticated AND not on `/login` → redirect to `/login`
- If authenticated AND on `/login` → redirect to role dashboard
- If authenticated AND accessing wrong role's pages → redirect to own dashboard

Role-based redirects:
- Retailer → `/retailer/products`
- Wholesaler → `/wholesaler/dashboard`
- Delivery → `/delivery/orders`

### 5. Build "No Access" Page

**File**: `pages/no-access.vue`

Shown when a phone number logs in but has no user document in Firestore. This means the wholesaler hasn't added them yet.

Content:
- "Your account is not registered"
- "Please contact your wholesaler to get access"
- Phone number displayed
- Logout button

### 6. reCAPTCHA Setup

Firebase Phone Auth requires reCAPTCHA verification to prevent abuse. Two options:

**Option A: Invisible reCAPTCHA** (recommended)
- Add a hidden div with id `recaptcha-container` on the login page
- Firebase handles the verification automatically
- User doesn't see anything

**Option B: Visible reCAPTCHA**
- Shows a checkbox "I'm not a robot"
- More intrusive but clearer

For MVP, use invisible reCAPTCHA.

## Edge Cases

| Scenario | Handling |
|----------|---------|
| Invalid phone number | Client-side validation before sending |
| OTP expired (after 60s) | Show error, enable "Resend" |
| Wrong OTP entered | Firebase returns error, show message |
| User blocked (`active: false`) | Check after login, redirect to no-access |
| Multiple tabs open | Firebase handles session sync |
| App refresh during OTP wait | User needs to restart login flow |
| Phone number not in India | Restrict to +91 prefix |

## Security Notes

- Never store OTP in the app. Firebase handles it.
- Don't expose Firebase Auth error codes directly to users (they leak info). Map them to friendly messages.
- Rate limiting is handled by Firebase (SMS abuse protection).
- Test phone numbers in development bypass reCAPTCHA and real SMS.
