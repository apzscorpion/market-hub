# 3.4 — Voice-to-Table Pipeline

## What We're Building

The complete flow that connects speech-to-text (3.1), parser (3.2), and the order table (Phase 2). When the user finishes speaking, the parsed items appear as rows in the order table.

## The Pipeline

```
Audio → [Web Speech API] → Text
  → [Text Cleaner] → Cleaned Text
    → [Voice Parser] → Parsed Items [{text, qty}]
      → [Product Matcher] → Matched Items [{product, qty, confidence}]
        → [Table Populator] → Order Table rows
```

## Step-by-Step

### Step 1: User Stops Speaking

When `recognition.onfinalresult` fires (or user taps "Stop"):
- Capture the full final transcript
- Pass to the pipeline

### Step 2: Clean the Text

Remove fillers, normalize whitespace, normalize Unicode (from 3.2/3.3).

### Step 3: Parse into Items

Voice parser extracts `[{text, quantity}]` (from 3.2).

### Step 4: Match Products

This is the new piece. Take each parsed item's `text` and find the matching product in the database.

```typescript
interface MatchedItem {
  parsedText: string          // what the user said
  quantity: number
  product: Product | null     // matched product (null if no match)
  confidence: 'high' | 'medium' | 'low' | 'none'
  alternatives: Product[]    // other possible matches (for disambiguation)
}
```

Use the matching algorithm from Phase 2.3 (nickname/alias matching):

- **High confidence**: Exact nickname or alias match
- **Medium confidence**: Starts-with or contains match (single result)
- **Low confidence**: Multiple products match (ambiguous)
- **None**: No product matches at all

### Step 5: Populate the Table

For each matched item:

| Confidence | Action |
|-----------|--------|
| High | Add row directly. Product cell shows the product name. Green indicator. |
| Medium | Add row with product selected. Yellow indicator — "Verify this match". |
| Low | Add row with dropdown open showing alternatives. Orange indicator — "Choose the correct product". |
| None | Add row with product cell showing the original text in red. User must manually search and select. |

## Product Matcher Implementation

**File**: `utils/productMatcher.ts`

```typescript
function matchParsedItems(
  parsedItems: ParsedItem[],
  products: Product[]
): MatchedItem[] {
  return parsedItems.map(item => {
    const results = matchProducts(item.text, products) // from Phase 2.3

    if (results.length === 0) {
      return { ...item, product: null, confidence: 'none', alternatives: [] }
    }

    if (results[0].score <= 1) {
      // Exact or starts-with nickname match
      return {
        parsedText: item.text,
        quantity: item.quantity,
        product: results[0].product,
        confidence: 'high',
        alternatives: results.slice(1, 4).map(r => r.product),
      }
    }

    if (results.length === 1 || results[0].score < results[1].score) {
      return {
        parsedText: item.text,
        quantity: item.quantity,
        product: results[0].product,
        confidence: 'medium',
        alternatives: results.slice(1, 4).map(r => r.product),
      }
    }

    // Multiple equally-scored matches
    return {
      parsedText: item.text,
      quantity: item.quantity,
      product: results[0].product,
      confidence: 'low',
      alternatives: results.slice(0, 4).map(r => r.product),
    }
  })
}
```

## Appending vs Replacing

When the user uses voice on a table that already has rows:
- **Append mode** (default): Voice results are added below existing rows
- If a voice result matches a product already in the table: increment the quantity instead of adding a duplicate row

This lets users build orders in multiple voice passes: "5 vanilla and 2 choco" → reviews → speaks again → "10 strawberry and 3 mango"
