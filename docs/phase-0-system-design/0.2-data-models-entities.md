# 0.2 — Data Models & Entities

## What We're Building

The complete Firestore database schema. Every collection, every document field, every relationship. This is the foundation everything reads from and writes to.

## Firestore Structure

Firestore is a document database. No tables, no joins. Think in terms of **collections** and **documents**.

---

### Collection: `users`

Each document represents one user in the system.

```
users/{userId}
├── name: string              — "Rajan's Store"
├── phone: string             — "+919876543210"
├── role: string              — "retailer" | "wholesaler" | "delivery"
├── active: boolean           — true (false = blocked)
├── address: string           — "Market Road, Thrissur" (optional)
├── fcmTokens: string[]       — push notification tokens (multiple devices)
├── preferredLanguage: string — "en" | "ml" (default: "ml")
├── createdAt: timestamp
└── updatedAt: timestamp
```

**Document ID**: Use Firebase Auth UID as the document ID. This makes security rules trivial (`request.auth.uid == resource.id`).

**Indexes needed**: `role` (to query all retailers, all delivery boys)

### Collection: `products`

Each document represents one product the wholesaler sells. Products are classified across 5 dimensions for rich filtering and discovery.

```
products/{productId}
├── name: string              — "Vanilla Ice Cream Big Pack 500ml"
├── nickname: string          — "Vanilla Big"
├── aliases: string[]         — ["vanilla", "vanila", "വാനില", "വാനിലാ", "van big"]
│
├── classification: {         — 5-dimensional product classification
│     typeId: string          — "family-pack" (cone/bar/cup/scoop/family-pack/bulk-pack/party-pack/tub/popsicle)
│     flavor: string          — "vanilla" (vanilla/chocolate/strawberry/mango/butterscotch/pista/mixed-fruit/tender-coconut/custom)
│     variant: string         — "premium" (standard/premium/sugar-free/kids-special/seasonal/limited-edition)
│     packagingType: string   — "box" (single/box/carton/loose)
│     temperatureSensitivity: string — "standard" (quick-melt/standard/long-hold)
│   }
│
├── categoryId: string        — reference to categories document
├── subcategoryId: string     — reference to subcategory
├── unitType: string          — "box" | "piece" | "pack" | "kg" | "litre"
├── unitsPerPackage: number   — 20 (e.g., 20 pieces per box)
├── price: number             — 250 (in rupees)
├── images: string[]          — Firebase Storage URLs
│
├── stock: {                  — stock tracking
│     quantity: number        — current stock count (in unitType units)
│     lowStockThreshold: number — alert when below this (e.g., 10)
│     lastUpdated: timestamp  — when stock was last changed
│   }
│
├── badges: {                 — product discovery flags
│     isNew: boolean          — show "New" badge (auto-expires after 14 days or manual toggle)
│     isTrending: boolean     — show "Trending" badge
│     isRecommended: boolean  — show "Recommended" badge (wholesaler's pick)
│   }
│
├── active: boolean           — true (false = hidden from catalog)
├── sortOrder: number         — for manual ordering in catalog
├── createdAt: timestamp
└── updatedAt: timestamp
```

**Why `classification` is an embedded object**: All classification fields are always loaded with the product and queried together. No need for a separate collection.

**Why `stock` is embedded**: Stock is tightly coupled to the product. Separate collection would mean extra reads on every product display.

**Why `badges` is embedded**: Same as stock — always needed when displaying the product.

**Why `aliases` is an array**: Voice recognition will produce variations. "Vanilla" might come as "vanila", "vanilla", "വാനില". We store all known variations so the parser can fuzzy-match against them.

**Indexes needed**: `categoryId` + `active`, `classification.typeId`, `classification.flavor`, `aliases` (array-contains for matching), `badges.isNew`, `stock.quantity` (for low stock queries)

### Collection: `productTypes`

Lookup collection for the classification system. Seeded at setup time.

```
productTypes/{typeId}
├── name: string              — "Family Pack"
├── nameML: string            — "ഫാമിലി പാക്ക്"
├── sortOrder: number
└── active: boolean
```

Similarly, collections for `flavors`, `variants` if they need to be dynamic. Or store as a single `classificationOptions` document with arrays — simpler for small lists.

```
classificationOptions/config
├── types: [{ id, name, nameML }]
├── flavors: [{ id, name, nameML }]
├── variants: [{ id, name, nameML }]
├── packagingTypes: [{ id, name, nameML }]
└── temperatureLevels: [{ id, name, nameML }]
```

Using a single config document avoids multiple reads and works well since these lists change rarely.

### Collection: `categories`

Flat list of categories. Subcategories are stored inline.

```
categories/{categoryId}
├── name: string              — "Ice Cream Cones"
├── nameML: string            — "ഐസ്ക്രീം കോൺ" (Malayalam)
├── sortOrder: number         — display order
├── subcategories: [          — embedded array
│   {
│     id: string              — "sub_vanilla_cones"
│     name: string            — "Vanilla Cones"
│     nameML: string          — "വാനിലാ കോൺ"
│     sortOrder: number
│   }
│ ]
├── createdAt: timestamp
└── updatedAt: timestamp
```

**Why subcategories are embedded**: They're small, always loaded with the category, and rarely change. A separate collection would just add unnecessary reads.

### Collection: `orders`

Each document represents one order placed by a retailer.

```
orders/{orderId}
├── retailerId: string        — reference to users document
├── retailerName: string      — denormalized for display ("Rajan's Store")
├── retailerPhone: string     — denormalized for quick contact
├── items: [                  — array of order line items
│   {
│     productId: string
│     productName: string     — denormalized (so order makes sense even if product changes)
│     nickname: string        — denormalized
│     quantity: number
│     unitType: string
│     price: number           — price at time of order
│     lineTotal: number       — quantity × price (denormalized for display)
│   }
│ ]
├── status: string            — "pending" | "accepted" | "shipped" | "delivered" | "cancelled"
├── assignedDeliveryId: string — null until assigned
├── assignedDeliveryName: string — denormalized
├── totalAmount: number       — sum of all lineTotal values
├── notes: string             — optional retailer notes
├── adminNotes: string        — internal wholesaler notes (not visible to retailer)
├── orderSource: string       — "manual" | "voice" (how the order was created)
├── invoiceId: string         — reference to invoices collection (null until invoice generated)
├── createdAt: timestamp
├── updatedAt: timestamp
└── statusHistory: [          — audit trail
    {
      status: string
      changedBy: string       — userId
      changedAt: timestamp
      note: string            — optional note about the change
    }
  ]
```

**Why denormalize names**: In Firestore, you can't join. If you only store `retailerId`, you need a second read to show the retailer name on the order list. Denormalizing avoids this. The trade-off is keeping it in sync — but names rarely change.

**Why `statusHistory`**: You need to know WHEN an order was accepted, shipped, delivered. A single `status` field only tells you the current state.

**Why `lineTotal`**: Avoids recalculating on every display. One-time cost at write time.

**Indexes needed**: `retailerId` + `createdAt` (retailer's order history), `status` (filter orders), `assignedDeliveryId` + `status` (delivery's orders), `createdAt` (date range queries)

### Collection: `invoices`

Auto-generated when an order is confirmed. This is the business document.

```
invoices/{invoiceId}
├── invoiceNumber: string     — sequential: "INV-2026-0001"
├── orderId: string           — reference to the order
├── retailerId: string
├── retailerName: string
├── retailerPhone: string
├── retailerAddress: string
├── wholesalerName: string    — from system settings
├── wholesalerAddress: string
├── wholesalerPhone: string
├── wholesalerGST: string     — GST number (if applicable)
├── items: [                  — snapshot of order items (frozen at invoice time)
│   {
│     productName: string
│     nickname: string
│     quantity: number
│     unitType: string
│     price: number
│     lineTotal: number
│   }
│ ]
├── subtotal: number          — sum of all lineTotal
├── taxRate: number           — e.g., 0.05 for 5% GST (0 if no tax)
├── taxAmount: number         — subtotal × taxRate
├── total: number             — subtotal + taxAmount
├── status: string            — "generated" | "sent" | "paid"
├── generatedAt: timestamp
├── paidAt: timestamp         — null until paid
└── pdfUrl: string            — Firebase Storage URL (optional, generated on demand)
```

**Why a separate collection from orders**: The invoice is a legal/financial document. It should be immutable once generated — even if the order is later modified or cancelled. The invoice captures the state at confirmation time.

**Invoice numbering**: Sequential numbering is important for accounting. Use a Firestore counter document or Cloud Function to ensure uniqueness.

### Collection: `retailerStats` (Computed)

Pre-computed statistics per retailer. Updated when orders change. Avoids expensive aggregation queries at read time.

```
retailerStats/{retailerId}
├── totalOrders: number
├── totalSpent: number
├── averageOrderValue: number
├── lastOrderDate: timestamp
├── frequentProducts: [       — top 10 most ordered products
│   {
│     productId: string
│     productName: string
│     totalQuantity: number   — cumulative quantity ordered
│     orderCount: number      — how many orders included this product
│   }
│ ]
└── updatedAt: timestamp
```

**Why pre-compute**: Querying all orders to find "frequent items" on every page load is expensive. Better to update stats when orders are placed/completed and read the cached result.

---

### Collection: `activityLogs` (Phase 5)

Audit trail for admin actions.

```
activityLogs/{logId}
├── action: string            — "order_created" | "status_changed" | "product_added" | ...
├── performedBy: string       — userId
├── performedByName: string   — denormalized
├── performedByRole: string
├── targetType: string        — "order" | "product" | "user" | "invoice"
├── targetId: string          — the document that was affected
├── details: object           — varies by action type
├── createdAt: timestamp
```

---

### Collection: `voiceLogs` (Phase 3)

Logs of voice ordering sessions for improving accuracy.

```
voiceLogs/{logId}
├── userId: string
├── language: string          — "en-IN" | "ml-IN"
├── rawTranscript: string     — what the speech API returned
├── cleanedText: string       — after filler removal and normalization
├── parsedItems: [            — what the parser extracted
│   { text: string, quantity: number }
│ ]
├── matchedProducts: [        — what was matched
│   { productId: string, confidence: string, parsedText: string }
│ ]
├── userCorrections: [        — what the user changed
│   { original: string, correctedTo: string }
│ ]
├── createdAt: timestamp
```

## Relationships Diagram

```
users (retailer) ──┐
                   ├──→ orders ──→ invoices
users (delivery) ──┘       │
                           ▼
                      products ──→ categories
                           │
                           ▼
                  classificationOptions

retailerStats ←── computed from orders + products
voiceLogs ←── logged during voice ordering
activityLogs ←── logged on all admin actions
```

- A **retailer** creates many **orders**
- Each **order** generates one **invoice**
- An **order** contains many **products** (as items array)
- Each **product** belongs to one **category** and has **classification** attributes
- A **delivery boy** is assigned to many **orders**
- **retailerStats** is a computed cache per retailer
- The **wholesaler** manages everything

## Firestore Read/Write Patterns

| Operation | Query | Frequency |
|-----------|-------|-----------|
| Retailer views products | `products` where `active == true`, ordered by `category` | Very high |
| Retailer views frequent items | `retailerStats/{uid}` → `frequentProducts` | High |
| Retailer views own orders | `orders` where `retailerId == uid`, ordered by `createdAt desc` | High |
| Retailer views stock | Part of product read (`stock.quantity`) | High |
| Wholesaler views all orders | `orders` ordered by `createdAt desc`, filtered by `status` | High |
| Wholesaler views low stock | `products` where `stock.quantity <= stock.lowStockThreshold` | Medium |
| Delivery views assigned orders | `orders` where `assignedDeliveryId == uid` and `status in [accepted, shipped]` | Medium |
| Voice parser matches product | `products` where `aliases` array-contains `"vanilla"` | High (during voice) |
| Generate invoice | Single document write to `invoices` + update `orders.invoiceId` | On order confirm |
| Admin edits product | Single document read + write | Low |

## Data Validation Rules

These will be enforced in Firestore security rules AND client-side:

- `quantity` must be > 0
- `price` must be >= 0
- `status` transitions must follow: pending → accepted → shipped → delivered (no skipping)
- `role` must be one of: "retailer", "wholesaler", "delivery"
- `phone` must match Indian format: +91XXXXXXXXXX
- `items` array must have at least 1 item to create an order
- `unitType` must be one of the allowed values
- `classification.typeId` must reference a valid product type
- `stock.quantity` must be >= 0
- `invoice.invoiceNumber` must be unique (enforced by Cloud Function)
- `badges` fields must be boolean

## Migration Strategy

There is no migration — this is a new system. But keep in mind:
- If you change the schema later, you need to update ALL existing documents
- Firestore has no schema enforcement — the app code IS the schema
- Use TypeScript interfaces to enforce document shapes in code
- The `classification`, `stock`, and `badges` fields are optional during Phase 1 (set defaults) but required from Phase 4 onward
