# 0.3 — Tech Stack & Architecture

## What We're Deciding

Every technology choice for the project, with reasoning. No "because it's popular" — every pick must earn its place.

---

## Frontend: Nuxt 3

**What it is**: A Vue 3 meta-framework with SSR, file-based routing, auto-imports, and a module ecosystem.

**Why Nuxt 3 over plain Vue 3**:
- File-based routing eliminates manual route config
- Auto-imports for composables and components reduce boilerplate
- SSR support out of the box (useful for SEO if needed later)
- Built-in middleware system for auth guards
- Nuxt modules for common needs (PWA, i18n)
- You already know it

**Why not React/Next.js**: Personal preference and existing skill. Both are capable. Pick what you ship fastest with.

**Key Nuxt features we'll use**:
- `pages/` directory for routing
- `middleware/` for auth guards
- `composables/` for shared logic (useAuth, useCart, useProducts)
- `components/` with auto-import
- `layouts/` for role-based layouts
- `plugins/` for Firebase initialization

---

## State Management: Pinia

**What it is**: Official Vue 3 state management library (replacement for Vuex).

**Why Pinia**:
- Official, first-party support
- TypeScript-first
- No mutations — just state, getters, actions
- Devtools integration
- Works perfectly with Nuxt 3 via `@pinia/nuxt`

**Stores we'll create**:
- `useAuthStore` — current user, role, auth state
- `useProductStore` — product catalog, categories
- `useCartStore` — current order being built (Phase 1) → later replaced by table state (Phase 2)
- `useOrderStore` — order history, order status

---

## Backend: Firebase

**What it is**: Google's Backend-as-a-Service. No server to manage.

**Services we'll use**:

| Service | Purpose | Free Tier Limit |
|---------|---------|----------------|
| **Firebase Auth** | Phone OTP login | 10K verifications/month |
| **Cloud Firestore** | Database | 50K reads, 20K writes, 20K deletes per day |
| **Firebase Storage** | Product images | 5GB storage, 1GB/day download |
| **Firebase Hosting** | Deploy the Nuxt app | 10GB storage, 360MB/day transfer |
| **Cloud Functions** | Server-side logic (later) | 2M invocations/month |

**Why Firebase over Supabase/custom backend**:
- Zero server management
- Phone auth built-in (critical for Kerala market)
- Firestore's real-time updates are perfect for order status tracking
- Free tier is generous enough for launch
- Scales automatically if the product takes off

**Why NOT a custom API (Express/Fastify)**:
- You don't need it yet
- Firebase security rules handle authorization
- Adding a server doubles the infrastructure to maintain
- If needed later, Cloud Functions fill the gap

**Trade-offs**:
- Vendor lock-in (acceptable for this stage)
- Firestore query limitations (no full-text search, no joins)
- Read/write costs can surprise you at scale

---

## Voice: Web Speech API → Google Cloud Speech-to-Text

**Phase 3 start**: Browser's built-in Web Speech API
- Free
- Works offline on some devices
- Malayalam support varies by device/browser

**Phase 3 upgrade**: Google Cloud Speech-to-Text
- Better accuracy for Malayalam
- Costs money ($0.006 per 15 seconds)
- Requires Cloud Functions to proxy the API call

The parser (speech text → structured order) is custom code regardless of which STT engine is used.

---

## Styling: Tailwind CSS

**Why**: Utility-first, fast to iterate, consistent design without writing custom CSS files. Pairs well with Nuxt via `@nuxtjs/tailwindcss` module.

**Component library**: None to start. Build custom components. If complexity grows, consider Headless UI or Radix Vue for accessible primitives (dropdowns, modals, etc.)

---

## Language Support: @nuxtjs/i18n

For Malayalam + English support. Handles:
- UI text translation
- Route localization (optional)
- Language switching

Translation files stored in `locales/en.json` and `locales/ml.json`.

## Application Architecture

```
┌──────────────────────────────────────────┐
│                 Nuxt 3 App                │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │  Pages   │  │Components│  │Layouts │ │
│  │(routing) │  │  (UI)    │  │(shells)│ │
│  └────┬─────┘  └────┬─────┘  └───┬────┘ │
│       │              │            │      │
│  ┌────▼──────────────▼────────────▼────┐ │
│  │          Composables                │ │
│  │  useAuth / useProducts / useOrders  │ │
│  └────────────────┬────────────────────┘ │
│                   │                      │
│  ┌────────────────▼────────────────────┐ │
│  │          Pinia Stores               │ │
│  │  authStore / productStore / etc.    │ │
│  └────────────────┬────────────────────┘ │
│                   │                      │
│  ┌────────────────▼────────────────────┐ │
│  │        Firebase SDK (client)        │ │
│  │  Auth / Firestore / Storage         │ │
│  └─────────────────────────────────────┘ │
└──────────────────────────────────────────┘
                    │
                    ▼
          ┌─────────────────┐
          │  Firebase Cloud  │
          │  Auth + Firestore│
          │  + Storage       │
          └─────────────────┘
```

**Data flow**:
1. User interacts with a **Page** or **Component**
2. Component calls a **Composable** function
3. Composable reads/writes via **Pinia Store** (for cached state) or directly to **Firebase** (for writes)
4. Firebase returns data → Pinia updates → Vue reactivity updates the UI

**Why composables + stores (not just one)**:
- Stores hold state (the data)
- Composables hold logic (how to fetch, validate, transform)
- Components stay thin — just render and call composable methods

## Deployment Strategy

**Development**: `npm run dev` locally, Firebase Emulator Suite for testing without hitting production.

**Staging**: Deploy to Firebase Hosting preview channel for testing.

**Production**: Deploy to Firebase Hosting via `firebase deploy`.

Later (Phase 9): Set up GitHub Actions for automated deployment on push to `main`.
