# 0.1 — User Roles & Permissions

## What We're Building

The permission system that controls who can do what. Every API call, every page, every button visibility depends on this.

## The Three Roles

### Retailer (Shop Owner)

The primary user. They place orders to the wholesaler.

**Can do:**
- Log in with phone number
- View the product catalog (only products the wholesaler has published)
- Create new orders (manual or voice)
- Edit their own pending orders
- Cancel their own pending orders
- View their order history
- View order status (pending → accepted → shipped → delivered)

**Cannot do:**
- See other retailers' orders
- Modify product catalog
- Change order status (only wholesaler/delivery can)
- Access admin panel
- See pricing tiers of other retailers (future: per-retailer pricing)

### Wholesaler (Admin)

The system owner. They control products, pricing, and order flow.

**Can do:**
- Everything a retailer can do (for testing)
- Add / edit / delete products
- Manage categories and subcategories
- Set product nicknames and aliases
- View ALL orders from ALL retailers
- Accept or reject orders
- Modify order items (adjust quantities, swap products)
- Assign orders to delivery personnel
- Add / remove / block users (retailers and delivery)
- View activity logs
- Access full admin panel

**Cannot do:**
- Nothing. This is the super-admin role.

### Delivery Boy

Fulfills physical delivery of orders.

**Can do:**
- Log in with phone number
- View orders assigned to them
- Mark order as "picked up"
- Mark order as "delivered"
- View delivery history

**Cannot do:**
- View unassigned orders
- Modify order contents
- Access product management
- Access user management
- Cancel orders

## Authentication Method

**Phone number + OTP via Firebase Auth.**

Why not email/password:
- Target users are small shop owners in Kerala
- Phone is universal, email is not
- OTP is familiar (they use it for UPI daily)

Flow:
1. User enters phone number
2. Firebase sends OTP
3. User enters OTP
4. Firebase creates/verifies user
5. App checks Firestore for user document → gets role
6. Route to appropriate dashboard

## How Roles Are Assigned

Roles are NOT self-selected. The wholesaler creates user accounts.

1. Wholesaler adds a new retailer (enters phone number + name + role)
2. A user document is created in Firestore with `role: "retailer"`
3. When that phone number logs in via OTP, the app reads the role and routes accordingly
4. If a phone number logs in but has no user document → show "Contact your wholesaler" screen

This prevents random signups. Only the wholesaler decides who gets access.

## Implementation Notes

- Store role in Firestore `users` collection, NOT in Firebase Auth custom claims (simpler for now, move to custom claims if needed for security rules later)
- Use Nuxt middleware to check role on every route change
- Create a `useAuth` composable that exposes `user`, `role`, `isRetailer`, `isWholesaler`, `isDelivery`
- All role checks happen both client-side (for UI) and in Firestore security rules (for data safety)
