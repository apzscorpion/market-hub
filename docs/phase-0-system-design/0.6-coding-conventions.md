# 0.6 — Coding Conventions

## What This Covers

Rules the codebase follows. Not suggestions — rules. Consistency makes everything faster to read, debug, and extend.

---

## File Naming

| Type | Convention | Example |
|------|-----------|---------|
| Pages | `kebab-case.vue` | `order-history.vue` |
| Components | `PascalCase.vue` | `OrderTable.vue` |
| Composables | `camelCase.ts` (prefixed with `use`) | `useAuth.ts` |
| Stores | `camelCase.ts` | `auth.ts` → exports `useAuthStore` |
| Utils | `camelCase.ts` | `voiceParser.ts` |
| Types | `camelCase.ts` | `order.ts` |
| Middleware | `camelCase.ts` | `auth.ts` |

## Component Structure

Every Vue component follows this order:

```vue
<template>
  <!-- template first, always -->
</template>

<script setup lang="ts">
// 1. imports (auto-imported ones don't need explicit import)
// 2. props & emits
// 3. composables (useAuth, useRoute, etc.)
// 4. reactive state (ref, reactive, computed)
// 5. functions
// 6. lifecycle hooks (onMounted, etc.)
// 7. watchers
</script>
```

No `<style>` blocks. Use Tailwind utility classes exclusively. If you need a custom style, put it in `assets/css/main.css`.

## TypeScript

- **Strict mode**: enabled in `tsconfig.json`
- **No `any`**: use proper types. If truly unknown, use `unknown` and narrow.
- **Interfaces over types**: for object shapes, use `interface`. Use `type` for unions and aliases.
- **Explicit return types**: on composables and util functions. Components don't need them.

## Composable Patterns

Every composable:
- Named with `use` prefix
- Returns an object (not array) for named destructuring
- Handles its own loading/error state

```typescript
export function useProducts() {
  const products = ref<Product[]>([])
  const loading = ref(false)
  const error = ref<string | null>(null)

  async function fetchProducts() {
    loading.value = true
    error.value = null
    try {
      // fetch from Firestore
    } catch (e) {
      error.value = (e as Error).message
    } finally {
      loading.value = false
    }
  }

  return { products, loading, error, fetchProducts }
}
```

## Store Patterns (Pinia)

```typescript
export const useAuthStore = defineStore('auth', () => {
  // Use setup syntax (composition API style)
  const user = ref<User | null>(null)
  const role = computed(() => user.value?.role ?? null)

  const isRetailer = computed(() => role.value === 'retailer')
  const isWholesaler = computed(() => role.value === 'wholesaler')
  const isDelivery = computed(() => role.value === 'delivery')

  async function login(phone: string) { /* ... */ }
  function logout() { /* ... */ }

  return { user, role, isRetailer, isWholesaler, isDelivery, login, logout }
})
```

## Error Handling

- **Never swallow errors silently.** Always log or surface them.
- **User-facing errors**: show via a toast/notification component (build a simple one)
- **Console errors**: use `console.error` with context: `console.error('[useOrders] Failed to fetch:', err)`

## Firebase Patterns

- Always import Firebase modular SDK (tree-shakeable):

```typescript
// correct — modular
import { collection, getDocs } from 'firebase/firestore'

// wrong — compat (bloated)
import firebase from 'firebase/app'
```

- Use `serverTimestamp()` for all timestamp fields:

```typescript
import { serverTimestamp } from 'firebase/firestore'

const order = {
  createdAt: serverTimestamp(),
  updatedAt: serverTimestamp(),
}
```

## Git Conventions

- **Branch naming**: `phase-X/feature-name` (e.g., `phase-1/auth-login`)
- **Commit messages**: imperative mood, short first line
  - `add phone OTP login flow`
  - `fix order status not updating`
  - `refactor product store to use composable pattern`
- **No commits with broken builds**. Run `npm run dev` before committing.

## What NOT to Do

- Don't install UI component libraries (Vuetify, PrimeVue, etc.) — build what you need with Tailwind
- Don't add Firebase Admin SDK to the client (it's server-only)
- Don't use Vuex (use Pinia)
- Don't use Options API (use Composition API with `<script setup>`)
- Don't store sensitive data in localStorage
- Don't skip TypeScript types ("I'll add them later" means never)
